### Added
- `docs/runbooks/`: six public operator runbooks (README, 00-deploy, 01-eif-upgrade, 02-key-rotation, 03-incident-response, 04-disaster-recovery, 05-operator-onboarding) per RFC §1.2 public/private split. M3-GA-gated and `qfc-core`-gated sections are explicitly marked "Pending" rather than written as if live; private counterparts live in `qfc-server-wallet-ops`.

- **M3-followup**: close the parse half of [D24](docs/m3-decisions.md#d24) — real COSE_Sign1 CBOR parsing in `verify_attestation`.
  - `qfc-enclave::cose`: new module wrapping the [`coset`](https://docs.rs/coset/) crate on top of [`ciborium`](https://docs.rs/ciborium/) (both pure-Rust, no FFI; matches the no-OpenSSL-in-the-enclave-attack-surface rule from RFC §1.5).
    - `parse_cose_sign1(bytes) -> CoseSign1Envelope` accepts both CBOR-tagged (production / AWS Nitro) and untagged (test) forms, owns the raw bytes for re-serialization, and surfaces structural errors via `CoseParseError::{MalformedEnvelope, MalformedPayload, MissingPayload, MissingField, WrongFieldType}`.
    - `extract_payload(&envelope) -> AttestationPayload` decodes the inner Nitro attestation-doc map exactly per the AWS spec (`module_id` / `timestamp` / `digest` / `pcrs` / `certificate` / `cabundle` / `public_key` / `user_data` / `nonce`). Unknown keys are tolerated for forward-compat.
    - `verify_cose_signature(&envelope, leaf_pubkey)` runs the RFC 8152 §4.4 `tbs_data` computation via `coset` and verifies the ed25519 signature against a supplied 32-byte leaf public key. The same code shape handles tampered signatures, tampered payloads, wrong keys, malformed inputs — every gate surfaces a typed `CoseVerifyError`.
    - `build_test_envelope(payload, &signer_sk)` helper (doc-hidden) lets tests construct synthetic envelopes signed with a known ed25519 key without hand-rolling CBOR.
  - `qfc-enclave::verify_attestation`:
    - New `SignatureKind { Mock, CoseSign1Ed25519, CoseSign1Es384 }` enum surfaces which path a `NitroAttestationDoc` should verify through. `signature_kind` field on `NitroAttestationDoc` is `#[serde(default)]` so documents serialized before this change deserialize with `signature_kind = Mock` — back-compat preserved.
    - `NitroAttestationDoc::mock(...)` constructor — explicit replacement for the old struct-literal mock-doc construction, hard-codes `signature_kind: SignatureKind::Mock`. M1/M2/M3 tests + `verify_mock_attestation` continue to work unchanged.
    - `NitroAttestationDoc::from_cose_sign1(bytes)` constructor parses real CBOR envelopes, populates the existing `NitroAttestationDoc` fields, narrows the inner-payload nonce to `[u8; 32]`, and labels the document `CoseSign1Ed25519` (or `CoseSign1Es384` if the COSE protected header announces `alg = -35`).
    - `NitroAttestationDoc::parse(bytes)` dispatcher: tries `from_cose_sign1` first, surfaces `MalformedCose` otherwise. Mock-format callers continue to construct via `mock(...)` (no signature material in the input bytes alone for mock).
    - `verify_attestation` dispatches on `signature_kind`: mock path runs the existing JSON round-trip + ed25519 check; `CoseSign1Ed25519` re-decodes via `parse_cose_sign1` + checks the parsed-payload mirror against the on-wire CBOR + verifies via `verify_cose_signature`; `CoseSign1Es384` routes through the stub. All other checks (PCR constraint, freshness window, cabundle non-empty, root-chain stub) are shared.
    - `verify_root_chain(leaf, cabundle, root) -> Result<(), AttestationVerifyError>` typed stub for the deferred AWS Nitro root-cert chain walker — see [D46](docs/m3-decisions.md#d46).
    - `verify_cose_signature_es384(envelope, leaf_cert)` stub returning `CoseVerifyError::AlgorithmNotImplemented` — see [D47](docs/m3-decisions.md#d47).
  - `qfc-enclave::lib`: new re-exports for `parse_cose_sign1`, `verify_cose_signature`, `verify_cose_signature_es384`, `CoseParseError`, `CoseSign1Envelope`, `CoseVerifyError`, `verify_root_chain`, `SignatureKind`.
  - New deps in `[workspace.dependencies]`: `coset = "0.4"` (pure-Rust COSE, Google + RustCrypto), `ciborium = "0.2"` (pure-Rust CBOR codec coset is built on; also used directly to encode/decode the inner attestation payload map). Both Apache-2.0 — no `deny.toml` change required.
  - Tests: 13 new `cose::tests` (envelope round-trip, signature tamper, payload tamper, wrong key, PCR map round-trip + ordering, truncated input, garbage bytes, empty bytes, missing-field rejection, ES384 stub surface, invalid-pubkey-length rejection, tagged envelope round-trip) + 8 new `verify_attestation::tests` covering the COSE end-to-end path (happy path, PCR mismatch, stale, signature tamper, payload-mirror tamper, malformed envelope, ES384 stub routing, back-compat default `signature_kind = Mock` deserialization). Workspace test count: 351 → 372 (+21). All M1/M2/M3 attestation tests untouched.
  - `docs/m3-decisions.md` extended with [D46](docs/m3-decisions.md#d46) (root-cert chain validation stays deferred; `verify_root_chain` lands as a typed stub) and [D47](docs/m3-decisions.md#d47) (ECDSA-P384 deferred; ed25519 ships first; `SignatureKind` makes the dispatch explicit).
  - Closes the parse half of M3 [D24](docs/m3-decisions.md#d24). The remaining M3-GA pieces tracked under D46 + D47 are scoped to a real-AWS PR: AWS Nitro root cert embed + cabundle chain walker, plus the ES384 verifier swap (`coset` parses ES384 envelopes today; only the curve plug changes).

- **gRPC API**: tonic-based gRPC surface alongside the existing HTTP API (RFC §10 decision #7).
  - Three protos under `crates/qfc-server-wallet/proto/` cover the M2/M3/M4 surfaces:
    - `wallet.proto` — `Wallet` service: `CreateWallet`, `GetWallet`, `Sign`, `GetAuditEvents`.
    - `approver.proto` — `Approver` service: `RegisterApprover`, `RevokeApprover`, `GetApprover`, `ListApprovers`, `CreateApproverSet`, `GetApproverSet`, `ListApproverSets`, `SubmitApproval`, `ListApprovals`.
    - `common.proto` — shared types: `SigningScheme`, `HashAlg`, `VmType`, `WalletStatus`, `ApproverStatus`, `ApprovalDecision`, `AuditKind`, `ApproverIdentity` (4-variant oneof), `Requester` (4-variant oneof), `SigningContext`, `SigningPayload` (4-variant oneof).
  - Wire conventions chosen to mirror the HTTP surface where they make sense and diverge where binary semantics win: ULIDs as strings (copy-paste friendly), keys / signatures / 32-byte hashes as raw `bytes` (no hex detour), timestamps as `int64` unix-ms, enums with explicit numeric values + an `_UNSPECIFIED` zero variant (fails loudly on default zero). JSON blobs (typed-data, audit details, attestation, `extra`) ride as UTF-8 `string` fields so the gRPC surface stays out of the structured-JSON business.
  - `qfc-server-wallet::grpc` module wires `WalletServer<WalletServiceImpl>` + `ApproverServer<ApproverServiceImpl>` directly onto `Arc<WalletService>`. **Zero logic duplication** — both transports adapt to the same handler core. `convert.rs` houses the domain ↔ proto conversion helpers (one direction at a time via `From` / `TryFrom`); `wallet.rs` and `approver.rs` are thin lower → call → raise translators.
  - Auth: `grpc::auth::ApiKeyInterceptor` reads the `x-api-key` metadata key on every RPC and validates against the same `Arc<HashSet<String>>` the HTTP middleware uses (same constant-time membership check via `subtle::ConstantTimeEq`). Missing / wrong key returns `tonic::Status::unauthenticated`.
  - Status-code mapping is consistent across transports: `BAD_REQUEST → INVALID_ARGUMENT`, `UNAUTHORIZED → UNAUTHENTICATED`, `FORBIDDEN → PERMISSION_DENIED`, `NOT_FOUND → NOT_FOUND`, `CONFLICT → ALREADY_EXISTS`, `UNPROCESSABLE_ENTITY → FAILED_PRECONDITION`, `INTERNAL_SERVER_ERROR → INTERNAL`. See `convert::map_service_error` / `map_registry_error` for the authoritative mapping.
  - `main.rs` now starts **both** servers concurrently from a single `Arc<AppState>`. HTTP binds `127.0.0.1:8088` by default (`QFC_SERVER_WALLET_HTTP_BIND` or back-compat `QFC_SERVER_WALLET_BIND`); gRPC binds `127.0.0.1:9090` (`QFC_SERVER_WALLET_GRPC_BIND`). Either can be disabled at startup via `QFC_SERVER_WALLET_DISABLE_HTTP` / `QFC_SERVER_WALLET_DISABLE_GRPC`. Both share a single graceful-shutdown future (SIGINT / SIGTERM race).
  - Reflection (`tonic-reflection`) is enabled by default behind a `reflection` cargo feature so `grpcurl` / `evans` / `bloomrpc` can introspect the schema without compiled stubs. Disable at runtime via `QFC_SERVER_WALLET_DISABLE_REFLECTION=1` or build with `--no-default-features` for prod images that want minimal surface area ([D48](docs/grpc-decisions.md#d48)).
  - Tests: 10 new (7 integration in `crates/qfc-server-wallet/tests/grpc_integration.rs` driving the full happy paths + the auth + error mappings via in-process tonic server on an ephemeral TCP port + auto-generated client stubs; 3 unit tests in `grpc/auth.rs` covering the metadata interceptor). E2E coverage: `Wallet::CreateWallet → GetWallet → Sign → GetAuditEvents` and `Approver::RegisterApprover → CreateApproverSet → GetApproverSet → SubmitApproval → ListApprovals → RevokeApprover → ListApprovers → ListApproverSets`. Workspace test count: 351 → 361 (+10). All four CI gates (test, clippy `-D warnings`, fmt, doc) plus `cargo deny check` clean.
  - New deps in `[workspace.dependencies]`: `tonic = "0.12"`, `tonic-build = "0.12"` (build-dep), `tonic-reflection = "0.12"`, `prost = "0.13"`, `prost-types = "0.13"`, `tokio-stream = "0.1"` (dev-dep only — needed to wrap a `TcpListener` as a tonic `Incoming` in the integration tests). Versions pinned to match the `tonic 0.12` line `opentelemetry-otlp 0.26`'s `grpc-tonic` feature already pulls in — avoids a duplicate-tonic build ([D50](docs/grpc-decisions.md#d50)).
  - `crates/qfc-server-wallet/build.rs` runs `tonic-build` over the three protos at compile time. Generated code lands in `OUT_DIR` (not committed) and is imported via `tonic::include_proto!("qfc.wallet.v1")`. Both server and client stubs are emitted; the client stubs are used internally by the integration tests but the crate does not publish a stand-alone client SDK ([D47](docs/grpc-decisions.md#d47)).
  - `#![forbid(unsafe_code)]` preserved everywhere (tonic generates safe code by default). `#![allow(clippy::result_large_err)]` scoped to each `grpc/*.rs` file with a one-line comment pointing to [D49](docs/grpc-decisions.md#d49) — `tonic::Status` is intrinsically ~176 bytes; boxing it would defeat the standard tonic ergonomics.
  - Docs: new `docs/grpc-api.md` (operator-facing — service map, wire conventions, auth, reflection quickstart, status-code mapping, env knobs); new `docs/grpc-decisions.md` (D46–D52 — proto-vs-HTTP-DTO separation, client-SDK deferral, reflection gating, version pinning, dual-server topology, audit-reader duplication trade-off).
  - **Not in this PR** (each documented in `docs/grpc-decisions.md` / `docs/grpc-api.md`): streaming RPCs (M2 surface is unary; audit-event tailing is a separate proposal); a published gRPC client SDK ([D47](docs/grpc-decisions.md#d47)); direct TLS (operators terminate at envoy / nginx, identical to the HTTP story).

- **M3-followup**: close the §3.4 GA loop — `PolicyServiceSigner` + hybrid verifier end-to-end wiring.
  - `qfc-policy`: new `PolicyServiceSigner` async trait (`sign_decision(decision, request_id, wallet_id, max_age_secs)` + `public_key()`) and `LocalPolicyServiceSigner` impl. The signer binds a `PolicyDecision` to a specific signing request + wallet, timestamps it via `time::OffsetDateTime::now_utc()`, and signs the byte-exact preimage produced by `SignedPolicyDecision::build_preimage` with an ed25519 identity key. Includes `LocalPolicyServiceSigner::generate(rng)` test helper plus 3 unit tests proving (a) `raw_payload` is verifier-canonical, (b) request/wallet binding survives, (c) generated keys are unique.
  - `qfc-enclave::enclaves::mock`: `MockEnclave` gains `with_policy_service_pubkey(Vec<u8>)` + `with_require_signed_decision(bool)` builders. When a pubkey is pinned, `sign_in_enclave` now invokes `HybridVerifier` BEFORE the SSS combine / curve sign — bringing mock backend parity with what the Nitro EIF boot binary does in production. The require-signed flag defaults to `false` for migration friendliness; the in-enclave verifier itself stays fail-closed per [D21](docs/m3-decisions.md#d21).
  - `qfc-enclave::enclave`: `EnclaveSignRequest` gains two additive optional fields — `wallet_ceilings: Option<WalletCeilings>` and `policy_signing_payload: Option<qfc_policy::SigningPayload>`. The verifier needs the structured payload to re-decode EVM transactions in-enclave (so it doesn't trust the host's pre-decoded `value`), and the wallet's projected hard ceilings to re-check `max_value_per_tx` / `contract_allowlist` / `chain_allowlist`. Both default to `None` so M1/M2 callers compile unchanged ([D37](docs/m3-decisions.md#d37)).
  - `qfc-server-wallet::WalletService`: new opt-in builder `with_policy_service_signer(Arc<dyn PolicyServiceSigner>)`. When wired, `sign` produces a `SignedPolicyDecision` after policy evaluation (Allow / RequireQuorum), emits the new `PolicyDecisionSigned` audit event, and threads the bundle into `EnclaveSignRequest.policy_decision` along with the wallet ceilings + structured payload. Default freshness window: `MAX_DECISION_AGE_SECS_DEFAULT = 60s` (constant in `service.rs`); operator-tunable per-call ([D33](docs/m3-decisions.md#d33), [D36](docs/m3-decisions.md#d36)). New `ServiceError::Internal(String)` variant absorbs signer-backend / clock failures.
  - `qfc-audit`: new `AuditKind::PolicyDecisionSigned` variant + stable kind byte `17`. Emitted between `SigningEvaluated` and `SigningAttempted` whenever the orchestrator signs a decision. The corresponding `AuditKindDto` (OpenAPI mirror) is extended. The hash-chain preimage format is unchanged for kinds `1..=16` ([D35](docs/m3-decisions.md#d35)).
  - Tests: 7 new E2E tests in `crates/qfc-server-wallet/tests/policy_service_signer_e2e.rs` covering (1) happy path (signer + matching enclave pubkey → sign succeeds + audit chain contains `PolicyDecisionSigned`), (2) wrong-key rejection (signer key A vs enclave pubkey B → `InvalidPolicyServiceSignature`), (3) stale decision (custom backdating signer → `DecisionStale`), (4) value-cap hard-ceiling violation (wallet cap 100, EVM payload value 200 → `ValueCapExceeded`), (5) back-compat (no signer wired → sign still works), (6) signer↔verifier preimage round-trip, (7) `ServiceError::Enclave` surface contract. Workspace test count: 315 → 322 (+7).
  - Docs: `docs/m3-decisions.md` extended with D33–D37 (PolicyServiceSigner trait shape, builder opt-in vs required-arg, kind byte assignment, default `max_age_secs`, additive `EnclaveSignRequest` ceiling fields).
  - Closes the retro `§3.4` gap end-to-end: the hybrid scheme is no longer a unit-tested island. The orchestrator → policy-service signer → enclave-side hybrid verifier path is integration-tested with the M1 `MockEnclave` and is ready to swap to `NitroEnclave` when the M3 GA PR lands.

- **M5**: post-quantum signing (ML-DSA-44 / -65 / -87) + QVM minimal decoder.
  - `qfc-enclave::signers::ml_dsa`: three signers (`MlDsa44Signer`, `MlDsa65Signer`, `MlDsa87Signer`) implementing the `Signer` trait over FIPS 204 (final). Built on the pure-Rust `ml-dsa` v0.1 crate (RustCrypto signatures workspace; no liboqs / no C FFI per RFC §1.5 and [D38](docs/m5-decisions.md#d38)). Public-key sizes are 1312 / 1952 / 2592 bytes; signature sizes are 2420 / 3309 / 4627 bytes. Secret material is the 32-byte FIPS 204 seed `xi` ([D39](docs/m5-decisions.md#d39)) so the SSS chunking layer treats PQ wallets like classical ones. `HashAlg::None` is the only accepted hash alg ([D40](docs/m5-decisions.md#d40)); any other value surfaces `SignerError::UnsupportedHash`. Signing uses the deterministic ML-DSA variant for reproducible tests.
  - `qfc-enclave::signer`: `signer_for_scheme` and `dispatch_signer` now route ML-DSA-44 / -65 / -87 to the real signers (M1 returned `NotImplemented`). `Result` return shape stays so future PQ schemes (SLH-DSA, Falcon) can land without churning callers ([D45](docs/m5-decisions.md#d45)).
  - `qfc-enclave::enclaves::mock`: `MockEnclave::generate_wallet` and `sign_in_enclave` now handle ML-DSA wallets end-to-end. The PQ branch allocates a 32-byte seed (vs the classical 64-byte BIP39 seed — [D44](docs/m5-decisions.md#d44)). HD paths on PQ wallets surface `EnclaveError::Derivation(SchemeNotHd)` at both generate-time and sign-time (RFC §9.1).
  - `qfc-policy::decoders::qvm`: M5 envelope-only QVM decoder per RFC §9.6 option (b) and [D41](docs/m5-decisions.md#d41). Parses `tx_type` / `chain_id` / `to` / `value` / `gas_limit` from the borsh `qfc-core::Transaction` shape; `data` is opaque (method-level / argument-level QVM policy deferred to M6 pending a `QvmCall` tx variant in `qfc-core`). Decoder is forward-compat: unknown tx-type discriminants map to `QvmTxType::Other(u8)` rather than rejecting, and trailing bytes after `data` are tolerated. Lives behind `QvmDecoder` (`VmDecoder` impl) the same as `EvmDecoder`; the policy evaluator can hold one of each.
  - `crates/qfc-server-wallet/tests/m5_multi_curve_quorum.rs`: integration test pinning the property that a wallet on **ML-DSA-65** can be authorised by two **ed25519** approvers. Drives the full sign flow through `WalletService::sign` (audit-NDJSON-tailing for the request_id, two signed approvals via `MemoryApprovalStore`, threshold trip, `MockEnclave` sign, FIPS 204 ML-DSA-65 signature externally verified against the wallet's master public key). Demonstrates that M4 D16's per-identity scheme dispatch already supports cross-curve quorum out of the box; no new code paths required.
  - `docs/design/cross-tee-quorum.md`: 1–2 page design covering the threat model (single-vendor TEE compromise — RFC §5.2 row 1), the `Vec<TeeBackendConstraint>` wallet-config shape, M-of-N attestation composition across {Nitro, SGX, TDX}, KMS implications, and the M-of-2 sequence diagram. Implementation **deferred to M6** ([D43](docs/m5-decisions.md#d43)) pending SGX + TDX backend acquisition + external review of the verifier composition rule.
  - **Out of M5 scope:**
    - Wallet migration tool (re-shard ed25519/secp256k1 wallet under ML-DSA scheme) — deferred to a separate `tools/wallet-migrate` deliverable ([D42](docs/m5-decisions.md#d42)). Migration is a customer-facing flow (new wallet has a different address per RFC §3.1 D4, needs operator approval flow + UX work); shipping the half-baked crypto piece risks operators running it without the surrounding ceremony.
    - Full QVM method-level decoder + WASM decoder — both gated on `qfc-core` (RFC §9.6).
  - New deps in `[workspace.dependencies]`: `ml-dsa = "0.1"` (pure-Rust ML-DSA, RustCrypto), `signature = "3.0"` (trait crate re-exported by `ml-dsa`), `borsh = "1.5"` (QVM decoder schema).
  - Tests: 17 new unit tests (3 size-pinned per ML-DSA level × `pk` + `sig`, plus round-trip + tamper + determinism + hash-alg rejection + wrong-seed-length + wrong-pk-length + scheme-accessor coverage; QVM decoder has 10 unit tests covering all four tx-type discriminants, opaque-data invariance, unknown-disc forward-compat, trailing-bytes tolerance, oversized-value saturation, and a fuzz-style random-bytes sweep). 3 new mock-enclave integration tests (one per ML-DSA level) for end-to-end generate→sign→verify. 1 new full-orchestrator integration test (`m5_multi_curve_quorum`) for the cross-curve quorum invariant.
  - `docs/m5-decisions.md` documents the 8 non-obvious calls (D38–D45) from this milestone.

- **M4**: real M-of-N quorum backend — registry, approval store, real notifiers, HTTP API.
  - `qfc-wallet-types`: two new ULID newtypes — `ApproverSetId` (an M-of-N group) and `ApproverId` (one registered approver record). Replaces the M1 placeholder where `qfc_policy::PolicyDecision::RequireQuorum` carried `ApprovalId` as a stand-in for the set id.
  - `qfc-quorum::registry` — `ApproverRegistry` async trait with `MemoryApproverRegistry` (in-memory `tokio::sync::RwLock<HashMap>`) and `PostgresApproverRegistry` (sqlx, embedded migrations `0002_approvers.sql`). Trait surface: `add_approver`, `revoke_approver` (soft delete), `get_approver`, `list_approvers_by_owner` (with `include_revoked` flag), `create_approver_set`, `get_approver_set`, `list_approver_sets`. Approver records carry the four-variant `ApproverIdentity` (Chain / External / Hardware / NestedWallet), curve, label, owner, optional webhook URL, lifecycle status, registration timestamp. Approver sets carry `(threshold, total)`, ordered member list, optional per-set `quorum_timeout_secs`. Cycle detection on `NestedWallet` membership at `create_approver_set` time via co-membership DFS with `MAX_NESTING_DEPTH = 3` hard ceiling. Per-set tx + advisory-lock-free design: validation done in application code inside a Postgres transaction (D33).
  - `qfc-quorum::store` — `ApprovalStore` async trait + `MemoryApprovalStore` + `PostgresApprovalStore`. Replay protection enforced at the DB layer via a `UNIQUE (request_id, approver_id)` constraint on the `approvals` table. Idempotent re-submission of the *same* approval payload returns `RecordOutcome::AlreadyRecorded`; a *different* payload from the same approver for the same request returns `ApprovalStoreError::DuplicateApproval`. Both backends agree on the outcome enum.
  - `qfc-quorum::approvers` — replaces `MockQuorumApprover` as the sole production `QuorumApprover` choice (mock remains for tests):
    - `WebhookApprover` — POSTs `application/json` notifications to a registered URL with `X-QFC-Signature: hex(HMAC-SHA256(secret, body))`. Per-request timeout configurable (default 5s).
    - `OnChainQfcEventApprover` — STUB: emits an `OnChainEvent` into a `tokio::sync::broadcast` channel. Real chain submission is gated on `qfc-core` integration not yet in the workspace (retro-m1-m2 §3.6); the trait shape + audit dispatch are wired so M5 can drop in the live submitter.
    - `HardwareApproverNotifier` — dispatch-only, same wire shape as `WebhookApprover` but distinct label `"hardware"` for ops dashboards. The hardware-key signing happens in the approver-side client; this type just notifies.
    - `OrchestratingApprover` — composes `Vec<Arc<dyn ApproverNotifier>>` + one `Arc<dyn ApprovalStore>`. `request_approval` fans out notifications concurrently; `collect_approvals` polls the store with a configurable backoff (default 50ms) AND wakes promptly on `notify_arrival` via an embedded `tokio::sync::Notify`. Surfaces the first reject; times out per the wallet/set-configured window.
  - `qfc-server-wallet::WalletService` integrates the new pieces additively:
    - `WalletService::new` now defaults to `MemoryApproverRegistry` + `MemoryApprovalStore`; production wires `PostgresApproverRegistry` / `PostgresApprovalStore` via the new `with_approver_registry` / `with_approval_store` builders. The `with_quorum_timeout` setter remains.
    - `sign` on `PolicyDecision::RequireQuorum` resolves the set from the registry, builds the `ApprovalRequest` from the registered identities, dispatches `request_approval` to the quorum, then awaits `collect_approvals` honouring the set's `quorum_timeout_secs` override. Audit emits the full transition set: `QuorumNotified`, `QuorumApprovalReceived(approver_key)`, `QuorumApprovalRejected(approver_key)`, `QuorumTimedOut`, and the new `QuorumThresholdReached` (kind byte 16).
    - `record_approval` now verifies the embedded signature, freshness, and approver-identity match against the registry record before persisting (D35); idempotent re-submit returns the `RecordOutcome` distinction so the HTTP layer can map to a stable `recorded: true|false`.
  - `qfc-server-wallet::api` (HTTP surface, all utoipa-annotated):
    - `POST /approvers` + `DELETE /approvers/{id}` + `GET /approvers/{id}` + `GET /approvers?owner=` (admin-gated).
    - `POST /approver-sets` + `GET /approver-sets/{id}` + `GET /approver-sets?owner=`. Cycle / depth / threshold violations map to `422 Unprocessable Entity`.
    - `POST /requests/{request_id}/approvals` (submit signed approval; verify-then-persist; 422 on signature/freshness failure, 409 on duplicate payload, 200 with `recorded: false` on idempotent re-submit) + `GET /requests/{request_id}/approvals`.
    - New schema DTOs: `ApproverIdentityDto` (all four variants, hex byte fields), `CreateApproverRequest`, `ApproverView`, `CreateApproverSetRequest`, `ApproverSetView`, `SubmitApprovalRequest`, `SubmitApprovalResponse`, `ApprovalView`, `ApprovalDecisionDto`, `ApproverStatusDto`.
  - `qfc-audit`: new `AuditKind::QuorumThresholdReached` variant + stable kind byte `16`. `PostgresAuditSink` decoder + the API DTO mirror updated. The hash-chain preimage format is unchanged for kinds `1..=15`; chains generated before M4 remain replay-verifiable.
  - `qfc-policy`: `PolicyDecision::RequireQuorum`'s `approver_set` field is retyped from `ApprovalId` to the new `ApproverSetId`. Internal-only churn (the field was a placeholder in M1).
