# `clients/` — approver-side reference clients

This directory ships the M4 RFC §7 deliverable: working reference
clients that an approver runs to receive `qfc-server-wallet` webhooks
and POST back signed approvals.

| Directory | Language | Purpose |
| --- | --- | --- |
| [`approver-rs/`](./approver-rs/) | Rust 1.88+ | Reference daemon that production deployments can fork as a starting point. Reuses `qfc-quorum::SignedApproval::signing_preimage` so the byte layout cannot drift from the server. |
| [`approver-ts/`](./approver-ts/) | Node 20 + TypeScript | Same wire contract in TypeScript. Lets web teams stand up an approver from a Node service without a Rust toolchain. Preimage layout pinned byte-exact against the Rust side via a generated golden-vector fixture. |

## Why two clients?

The server-side webhook + approval-submission API was complete after
M4, but no reference receiver existed. Every approver integrator would
have had to re-derive:

- the `X-QFC-Signature` HMAC verification rules,
- the canonical preimage byte layout (`SignedApproval::signing_preimage`),
- the `POST /requests/{request_id}/approvals` DTO shape.

We ship two reference implementations so the contract is unambiguous in
both ecosystems we expect approvers to live in:

1. **Treasury / vault operators** running a long-lived Rust daemon next
   to their HSM or KMS.
2. **Web teams** running a Node sidecar inside their existing service
   mesh, where adopting a Rust toolchain is a non-starter.

Neither client is part of the main `qfc-server-wallet` workspace —
they're intentionally standalone Cargo / npm projects so a fork can be
made without inheriting the entire wallet dependency tree.

## Security notes

> The reference clients sign software-resident keys. Production
> deployments **must** swap `signer_loader.rs` (Rust) / `signer.ts`
> (TypeScript) for a hardware-backed signer.

- **Webhook secret** (HMAC). Treat as a high-value secret — anyone
  with it can spoof signing requests to your approver. Mount via a
  file (`--webhook-secret @/path/to/file`), never via argv in
  production.
- **Approver signing key**. Even more sensitive — it authorises wallet
  signings. The reference impl reads 32 raw bytes from a file; harden
  to a YubiKey / KMS for real use.
- **`--auto-approve` is for demos only.** The reference daemon
  defaults to a refuse-everything fail-closed policy if no policy flag
  is passed. Use `--interactive` or build a sidecar HTTP control plane
  on top of `Processor`.

## Generating the cross-language test fixture

The TypeScript test fixture `clients/approver-ts/test/fixtures/preimage_golden.json`
is generated by a tiny Rust helper:

```sh
cd tools/gen-golden-vectors
cargo run --bin gen-golden-vectors
```

This must be re-run any time the Rust-side `signing_preimage` layout
changes; the TS preimage test will fail loudly otherwise.

See [`docs/clients-decisions.md`](../docs/clients-decisions.md) for the
non-obvious design calls (forkability, dependency choices, identity
override surface).
